## 1. 项目背景

本项目旨在建设一个支持团队知识沉淀、在线协作、AI 辅助问答和内容分发的知识库网站。系统需要支持用户创建知识库、编辑文档、多人实时协同、文档版本管理、基于知识库内容的 AI 问答，以及将知识库文档发布到微信公众号草稿箱或后续发布渠道。

第一阶段以后端 Java 为主，采用模块化单体架构。数据存储、缓存数据、全文搜索数据、向量数据均优先使用 PostgreSQL 承载。后续根据业务规模，可将 AI、协同编辑、发布任务等模块拆分为 Python 或 Go 服务。

**架构原则**: 第一阶段优先完成核心业务闭环，避免过早微服务化；所有异步能力通过任务表和状态机解耦。

## 2. 建设目标

### 2.1 业务目标

1. 支持创建和管理多个知识库。
2. 支持知识库文档的创建、编辑、保存、删除、归档和搜索。
3. 支持多人实时协同编辑。
4. 支持文档版本历史和回滚。
5. 支持基于知识库内容的 AI 问答。
6. 支持文档切片、Embedding 生成、向量检索。
7. 支持将文档发布到微信公众号草稿箱。
8. 支持用户、空间、知识库、文档多级权限控制。
9. 支持操作审计、任务追踪和异常重试。

### 2.2 技术目标

1. 后端第一阶段使用 Java + Spring Boot。
2. PostgreSQL 统一承担业务存储、缓存、全文检索、向量存储。
3. 向量能力使用 PostgreSQL pgvector 扩展。
4. 协同编辑使用 WebSocket。
5. 协同冲突解决优先采用 CRDT 方案。
6. 通过任务表和 Worker 机制处理异步任务。
7. 保留后续服务拆分能力，避免模块间强耦合。

## 5. 功能需求

### 5.1 用户与认证模块

#### 5.1.1 用户注册

用户可以通过邮箱、用户名和密码注册账号。

| 字段 | 必填 | 说明 |
|------|------|------|
| username | 是 | 用户名 |
| email | 是 | 邮箱 |
| password | 是 | 密码 |

1. 校验邮箱格式。
2. 校验邮箱是否已注册。
3. 密码加密存储。
4. 创建用户记录。
5. 返回用户基础信息。

```json
{
  "id": "uuid",
  "username": "kone",
  "email": "user@example.com"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|

| password | 是 | 密码 |

1. 根据邮箱查找用户。
2. 校验密码。
3. 生成登录态。
4. 返回 token 或 session 信息。

```json
{
  "access_token": "jwt-token",
  "refresh_token": "refresh-token",
  "expires_in": 7200
```

### 5.6 实时协同编辑模块

#### 5.6.1 建立协同连接

**协议**: WebSocket

```
/ws/v1/documents/{documentId}/collaboration
```

| 参数 | 说明 |
|------|------|
| token | 用户登录 token |
| client_id | 前端客户端 ID |
| document_id | 文档 ID |

1. 校验 token。
2. 校验用户是否有文档访问权限。
3. 加入文档协同房间。
4. 返回当前协同状态。
5. 广播用户上线状态。

#### 5.6.2 协同更新消息

```json
{
  "type": "doc_update",
  "document_id": "uuid",
  "client_id": "client-001",
  "update": "base64-binary",
  "seq": 1024
}
```

1. 校验连接权限。
2. 保存 CRDT update 到 document_collab_updates。
3. 广播 update 给同房间其他用户。
4. 根据策略触发快照合并。

#### 5.6.3 在线用户状态

```json
{
  "type": "presence",
  "document_id": "uuid",
  "user_id": "uuid",
  "username": "kone",
  "cursor": {
    "anchor": 100,
    "head": 120
  }
}
```

1. 支持在线用户列表。
2. 支持用户光标位置广播。
3. 支持用户选择区域广播。
4. 用户断开连接时广播离线状态。

#### 5.6.4 协同快照

为了避免每次打开文档都重放所有 update，需要定期生成协同快照。

| 触发条件 | 说明 |
|----------|------|
| 累计 N 条 update | 当增量日志达到阈值时合并 |
| 固定时间间隔 | 定时生成快照 |
| 房间无人在线 | 文档关闭后压缩状态 |
| 手动保存文档 | 保存时确保快照状态可恢复 |

1. 读取最近快照。
2. 应用后续 update。
3. 生成新的 state_binary。
4. 写入 document_collab_snapshots。
5. 可清理旧 update 或标记已归档。

### 5.8 AI 助手模块

#### 5.8.1 文档切片

文档保存后，系统异步将文档拆分为多个 chunk，用于 Embedding 和 RAG 检索。

1. 优先按标题层级切分。
2. 单个 chunk 建议控制在 300 到 800 tokens。
3. 保留 heading_path。
4. 保留 document_id、snapshot_id、kb_id。
5. 代码块、表格、引用块需要尽量保持完整。
6. 切片失败不影响文档保存，但需要记录任务失败原因。

#### 5.8.2 Embedding 生成

| 触发条件 | 说明 |
|----------|------|
| 文档新建 | 新文档首次生成索引 |
| 文档内容更新 | 内容变更后重建 |
| 文档回滚 | 回滚后基于新快照重建 |
| 手动重建索引 | 管理员或用户触发 |
| Embedding 模型变更 | 模型变化后需要重建 |

1. 读取 document_chunks。
2. 调用 Embedding Provider。
3. 将向量写入 document_chunk_embeddings。
4. 标记索引状态为 completed。
5. 失败时写入 job 错误信息并支持重试。

#### 5.8.3 知识库问答

```http
POST /api/v1/ai/chat
```

```json
{
  "conversation_id": "uuid",
  "workspace_id": "uuid",
  "kb_ids": ["uuid"],
  "question": "协同编辑为什么要用 CRDT?",
  "top_k": 8
}
```

1. 校验用户身份。
2. 校验用户对目标知识库的访问权限。
3. 将 question 转换为 query embedding。
4. 在 document_chunk_embeddings 中进行向量检索。
5. 同时执行全文检索。
6. 合并结果并重排。
7. 按文档权限过滤最终上下文。
8. 构造 Prompt。
9. 调用大模型生成回答。
10. 保存 ai_conversations 和 ai_messages。
11. 返回回答和引用来源。

```json
{
  "answer": "CRDT 可以在多人并发编辑时自动合并变更...",
  "citations": [
    {
      "document_id": "uuid",
      "title": "协同编辑协议",
      "chunk_id": "uuid",
      "quote": "CRDT 适合..."
    }
  ],
  "usage": {
    "prompt_tokens": 1200,
    "completion_tokens": 320,
    "total_tokens": 1520
  }
}
```

1. 校验文档访问权限。
2. 校验用户发布权限。
3. 锁定当前 document_snapshot。
4. 创建 publication_tasks。
5. 写入异步发布任务。
6. 返回任务 ID。

#### 5.9.3 文档转公众号格式

1. 标题转换。
2. 段落转换。
3. 图片上传并替换 URL。
4. 代码块转换为公众号兼容样式。
5. 表格转换为兼容 HTML。
6. 引用块转换。
7. 去除不兼容标签和脚本。
8. 生成最终 HTML。

#### 5.9.4 创建公众号草稿

1. 获取公众号 access_token。
2. 读取文档快照。
3. 解析文档中的图片。
4. 上传图片素材。
5. 替换正文图片地址。
6. 生成公众号 HTML。
7. 调用公众号草稿接口。
8. 保存 external_draft_id。
9. 更新 publication_task 状态为 draft_created。
10. 失败时记录错误并支持重试。

## 8. 非功能需求

### 8.1 性能要求

| 场景 | 要求 |
|------|------|
| 文档详情读取 | P95 小于 300ms |
| 知识库目录树读取 | P95 小于 500ms |
| 全文搜索 | P95 小于 800ms |
| AI 问答首 token | P95 小于 5s |
| WebSocket 消息广播 | 单房间 P95 小于 200ms |
| 文档保存 | P95 小于 500ms，不等待向量生成完成 |

### 8.2 可用性要求

1. 文档保存不能依赖 AI 服务可用性。
2. 公众号发布失败不能影响文档编辑。
3. AI 服务异常时返回明确错误信息。
4. 异步任务失败必须可重试。
5. 协同服务异常时，前端应可回退到普通保存模式。

### 8.3 安全要求

1. 密码必须加密存储。
2. 公众号 AppSecret 必须加密存储。
3. 所有文档访问必须进行权限校验。
4. AI 检索必须进行权限过滤。
5. 接口需要防止越权访问。
6. 上传文件需要校验 MIME 类型和大小。
7. 防止 XSS，文档 HTML 输出需要清洗。
8. WebSocket 连接必须鉴权。
9. 审计日志不可被普通用户修改。
10. 敏感配置不可明文返回前端。

### 8.4 可观测性要求

1. API 请求日志。
2. 错误日志。
3. 慢查询日志。
4. AI 调用日志。
5. AI token 用量。
6. 公众号接口调用日志。
7. WebSocket 在线连接数。
8. 异步任务成功率。
9. 异步任务失败原因。
10. PostgreSQL 查询性能指标。

## 15. 关键风险

| 风险 | 说明 | 建议 |
|------|------|------|
| PostgreSQL 同时承担太多职责 | 存储、缓存、向量、任务都放在 PostgreSQL，后期可能有性能压力 | 模块接口预留 Redis、MQ、对象存储替换能力 |
| 协同编辑复杂度高 | CRDT 状态、快照、重放、冲突排查复杂 | 第一阶段只支持单文档协同，优先稳定 |
| AI 越权风险 | 用户可能通过 AI 问答看到无权限内容 | 检索前后双重权限过滤 |
| 公众号接口不稳定 | 外部接口可能失败、限流或权限不足 | 发布任务异步化、状态化、支持重试 |
| 文档格式转换复杂 | 富文本到公众号 HTML 可能存在样式兼容问题 | 统一中间格式，先限制支持的内容块 |
| 向量检索质量不稳定 | 切片、Embedding、重排会影响回答质量 | 保留检索日志和评测数据 |
